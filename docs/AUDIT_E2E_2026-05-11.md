## 🔴 Findings — par sévérité

### Critical
*(aucun découvert en test non-authentifié — flows métier non testés)*

### High

#### H1 — CLS 0.1179 sur /annonces (Core Web Vital)
- **Mesure** : CLS 0.1179 ; seuil Google "Good" = ≤ 0.1.
- **Impact** : pénalité Google Search Console, expérience visiteur "bouge". Bloque potentiellement le bon référencement SEO.
- **Cause probable** : injection asynchrone d'éléments fixes (Navbar fixed, FiltersBar fixed, dossier banner) après hydratation React provoque un shift quand `MountedOnly` passe de fallback à vrai contenu.
- **Fix** : réserver les heights via `MountedOnly fallback` (déjà fait sur Navbar V74.3 avec `<div style={{ height: 72 }}>`), vérifier que FiltersBar a bien son spacer (V81.18 paddingTop:80 sur container).
- **Fichiers** : `app/components/MountedOnly.tsx`, `app/(public)/annonces/AnnoncesClient.tsx`

#### H2 — Focus trap manquant sur modales (héritage audit a11y V81.29)
- **Mesure** : `BottomNavSheet.tsx:306` et `Navbar.tsx:783` ont `role="dialog" aria-modal="true"` mais pas de focus trap.
- **Impact** : utilisateur clavier `Tab` sort du modal dans le DOM derrière → violation WCAG 2.1.2.
- **Fix** : ajouter un `onKeyDown` qui boucle `Tab`/`Shift+Tab` sur les éléments focusables internes au modal. Cf hook custom `useFocusTrap` à créer.
- **Fichiers** : `app/components/BottomNavSheet.tsx`, `app/components/Navbar.tsx`

#### H3 — Auth flows non testés E2E (gap audit)
- **Mesure** : phases 2-3-4 non exécutables sans login Google OAuth automatisé.
- **Impact** : 9 flows critiques métier non vérifiés : candidature, dossier envoyé, visite proposée/acceptée, bail généré, signature eIDAS, quittance, EDL.
- **Fix** : écrire suite Playwright avec credentials/cookies persistés (NextAuth Email magic link plutôt que Google) ou utiliser un compte test seedé en DB avec session JWT mock.
- **Fichiers** : créer `nestmatch/e2e/auth-flows.spec.ts`

### Medium

#### M1 — Tap targets < 44px sur Navbar desktop
- **Mesure** : 93 liens/boutons sur 111 mesurent moins de 44x44 (la plupart : liens Navbar `Annonces`/`Favoris`/`Messages` à 30px de haut).
- **Impact** : OK sur desktop souris (WCAG 2.5.5 AAA, pas AA). Bug potentiel sur tablet touch.
- **Fix** : ajouter `padding-y:8px` sur les liens Navbar pour atteindre 44px de hauteur tactile, sans changer la baseline visuelle.
- **Fichiers** : `app/components/Navbar.tsx`

#### M2 — Contraste `#8a8477` sur `#F7F4EF` borderline (3.4:1)
- **Cf audit a11y V81.29** — reporté dans cette session, à traiter en V82 design system migration.
- **Workaround court terme** : changer `km.muted` en `#6b6358` dans `km.tsx` (1 ligne, propage partout).

### Low

#### L1 — Console warning CSP `upgrade-insecure-requests` en report-only
- **Mesure** : visible dans la console : *"directive 'upgrade-insecure-requests' is ignored when delivered in a report-only policy"*.
- **Impact** : aucun (warning informational). En production le CSP est actif, pas report-only.
- **Fix** : retirer cette directive du `Content-Security-Policy-Report-Only` ou la déplacer sur la version active. Cf `next.config.js` ou middleware CSP headers.

#### L2 — `/api/favoris` lent (1.5s)
- **Mesure** : un fetch de `/favoris` met 1477 ms sur /annonces.
- **Impact** : retarde le rendu des cœurs favoris sur la liste. Pas bloquant car async.
- **Fix** : ajouter index Postgres sur `favoris(user_email, annonce_id)` si absent. Audit côté `app/api/favoris/route.ts`.

#### L3 — Annonce id=40 inexistante
- **Mesure** : `/annonces/40` retourne "Annonce introuvable" mais 200 (pas vrai 404).
- **Impact** : SEO mineur — Google peut indexer ces pages "404 soft". Aussi : pas d'`X-Robots-Tag: noindex` détecté.
- **Fix** : retourner `notFound()` Next.js depuis `app/(public)/annonces/[id]/page.tsx` quand l'annonce n'existe pas, pour vrai HTTP 404 + meta robots noindex.

## ⚠️ Limitations méthodologiques de cette session

1. **Google OAuth non automatisable** depuis Claude → flows Phase 2/3/4 non couverts. Solution : Playwright avec storageState seedé.
2. **Chrome MCP `resize_window`** ne resize que la fenêtre externe, `window.innerWidth` reste sur le viewport réel du moniteur → Phase 5 responsive impossible. Solution : DevTools device emulation programmatique (pas exposé par Chrome MCP).
3. **Chrome MCP screenshots flakys** (disconnect ~30% du temps) → tests visuels incomplets. Solution : retry logic ou Playwright headless.
4. **Pas d'accès aux logs Vercel/Sentry** en live → erreurs runtime non visibles depuis Chrome MCP.
5. **Pas de mesure LCP fiable** car `PerformanceObserver` setup après hydration → utiliser PageSpeed Insights ou Lighthouse CI pour mesures précises.
